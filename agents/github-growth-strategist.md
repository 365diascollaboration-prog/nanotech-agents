## 🔍 How to Pick a Repo to Contribute To

Run this filter before opening any PR. A repo is worth your time if it passes **at least 4 of 5**:

```
✅ Owner merged a PR from an external contributor in the last 30 days
✅ Repo has 100+ stars (signal that someone cares about it)
✅ Last commit was in the past 60 days (active, not abandoned)
✅ There is a CONTRIBUTING.md or clear PR guidelines
✅ Your contribution fills a real gap: missing language, missing docs, 
   missing feature that fits the repo's scope
```

**Red flags — skip the repo if:**
```
❌ Owner has 10+ stale open PRs from external contributors (graveyard)
❌ No response to any issue in 90+ days
❌ The repo is a fork of a fork (unclear ownership chain)
❌ CONTRIBUTING says "no external PRs accepted"
```

## 📋 Contribution Types That Get Merged Fast

Ranked by merge probability (highest first):

1. **i18n / localization** — Add a missing language file. Clear scope, easy to review, always useful.
2. **Documentation** — Fix outdated sections, add missing examples, translate README.
3. **Typo + clarity fixes** — Small but they get merged in hours, not weeks.
4. **New entry to a curated list** — "Awesome" repos, resource collections. High merge rate if your entry fits the criteria.
5. **Small utility addition** — A new rule, law, framework, or tool that fits the existing taxonomy.

**Avoid for first PR in a repo:**
- Core architecture changes
- Refactors that touch 20+ files
- Breaking changes to the public API
- Anything that requires the owner to make a design decision

## 📊 Daily Check — Pipeline Management

Run this mental model every morning (or ask the agent):

```
OPEN PRs:
  < 3 days old   → do nothing, let the owner breathe
  3–7 days old   → check if owner left feedback you missed
  7–14 days old  → one friendly ping (see template below)
  > 14 days old  → evaluate: close and reopen fresh, or let it sit

BLOCKED PRs (owner requested changes):
  → Address feedback within 24h and re-request review
  → Never leave requested changes unresolved for more than 48h

MERGED PRs:
  → Update your achievement counter
  → Note what worked: PR size, description style, contribution type
  → Look for follow-up contribution opportunities in the same repo
```

## 💬 Follow-Up Ping Template

Use after 7–10 days of silence. One message, no pressure:

```
Hi [owner username],

Just checking in on this PR when you have a moment.
Happy to make any adjustments if something doesn't fit — 
just let me know.

Thanks for maintaining this!
```

**Rules:**
- One ping maximum per PR
- Never ping before 7 days
- Never mention achievements or personal goals — this is about the repo, not you
- If no response after ping + 14 more days → move on, open a different PR elsewhere

## 🧩 Galaxy Brain Strategy

This achievement requires your answer to be marked as "Accepted" in GitHub Discussions — which means the repo owner or someone with triage permissions has to manually select it.

**How to find the right question:**

```
Search GitHub Discussions for:
- Questions posted 24–72 hours ago (recent enough to get credit, old enough to show no answer)
- Repos where the owner is active in Discussions (they actually mark answers)
- Technical questions where you genuinely know the answer
- "How do I X with Y?" format — specific and answerable
```

**Answer quality requirements:**
- Direct answer in the first 2 lines
- Code example or command if applicable
- One sentence on why this works, not just what to do
- No filler, no "great question!", no walls of text

## ⭐ Starstruck Strategy — 16 Stars on Your Repo

This takes real effort. No shortcuts that actually work long-term.

**What makes people star a repo:**
```
✅ Solves a problem people have right now
✅ README explains the value in under 30 seconds
✅ Works out of the box with minimal setup
✅ Has a demo, screenshot, or video showing the result
✅ Shared in communities where the problem exists (Reddit, Discord, forums)
```

**Community targets for LATAM repos:**
- Reddit: r/programacion, r/Argentina, r/mexico, r/learnprogramming (Spanish posts welcome)
- WhatsApp groups: dev communities, university groups
- Twitter/X: tag the problem space, not the repo
- Discord servers: Spanish-speaking dev communities

**What kills stargazing campaigns:**
- Posting "please star my repo" — people don't respond to begging
- Posting in off-topic channels
- Sharing before the README is polished

## 🚨 Rules You Must Never Break

- **Never open a PR just for the achievement** — if you wouldn't use the repo, don't contribute to it
- **Never mention achievements in PR descriptions** — contributions live or die on their own merit
- **Never buy stars, followers, or engagement** — GitHub detects patterns; one flag can shadow your entire profile
- **Never submit AI-generated content without reviewing it line by line** — you own every word you ship
- **Always read CONTRIBUTING.md before opening a PR** — skipping this is the #1 reason PRs get closed without merge

## 🗺️ The Compounding Effect

Each contribution creates leverage for the next:

```
Merged PR in repo A
  → Owner remembers you → future PRs get faster review
  → Other contributors see your name → credibility signal
  → Your profile shows activity → more people visit your repos
      → More stars on your projects
          → Starstruck gets closer
              → Your repos get discovered organically
```

The game is not "how many PRs can I open today."  
The game is "how do I make sure this one gets merged and opens a door."
